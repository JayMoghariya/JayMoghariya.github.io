Python

  • Decorator

    • Decorators are a powerful feature that allows you to modify the behavior of functions or methods using the @decorator syntax.
    • @staticmethod: This decorator is used to define a class method within a class. Class methods take the class (usually denoted as cls) as the first parameter and can access class-level variables.
    • @property: This decorator allows you to define a method as a getter for a class attribute.
    • @attribute.setter: This is used in combination with the @property decorator to define a method as a setter for the specified attribute.
    • @attribute.deleter: This is used in combination with the @property decorator to define a method as a deleter for the specified attribute.
    • @abstractmethod: This decorator is part of the abc (Abstract Base Classes) module and is used to define abstract methods in abstract classes. Abstract methods must be implemented by the concrete subclasses.
    • @functools.wraps: This is not directly related to class attributes but is commonly used with decorators to preserve the metadata of the original function, such as its name, docstring, and module.
    • @lru_cache: This decorator is part of the functools module and is used for memoization. It caches the results of a function and returns the cached result when the same arguments are passed again.

  • Compiler Vs. Interpretor

    • On a Broad Picture this is what happens:
      • Compiler takes all the code in one go. It converts it into machine language and feed it to computer.
      • Interpretor takes one instruction at a time. Convert into machine language and feed it to computer. After its execution it takes another instruction and so on. If error occurs in between it stops its execution.
      • In general, interpreters tend to be slower than compilers for certain types of tasks due to their different approaches to executing code.
      • Compiled code can indeed be faster than interpreted code, especially for tasks that involve a lot of mathematical computations or complex algorithms.
      • Just-In-Time (JIT) compilation used in some interpreters (e.g., PyPy for Python) can dynamically compile parts of the code to machine code, narrowing the performance gap between interpreters and compilers.

  • CPython vs PyPy

    • CPython is the default and most widely used implementation of Python. It is written in C and is the reference implementation of the language.
    • CPython includes a Global Interpreter Lock (GIL), which allows only one thread to execute Python bytecode at a time, limiting the performance of multi-threaded CPU-bound tasks.
    • PyPy is written in RPython (Restricted Python), and it includes a Just-In-Time (JIT) compiler, which dynamically compiles Python bytecode to machine code at runtime.
    • PyPy does not use the Global Interpreter Lock (GIL), which allows it to take better advantage of multiple CPU cores for multi-threaded CPU-bound tasks.
    • In general; PyPy is faster than CPython for CPU-bound tasks, but it is slower for I/O-bound tasks.

  • How Python Script executed?

    1. Tokenization: The Python interpreter reads the source code and breaks it down into individual tokens (keywords, identifiers, literals, etc.).
    2. Parsing: The tokens are organized into a syntax tree, known as the Abstract Syntax Tree (AST). This step checks the code for syntax errors and creates a structured representation of the program.
    3. Compilation: The AST is compiled into bytecode, which is a lower-level representation of the code that the Python interpreter can understand and execute.
    4. Interpretation: The Python interpreter reads the bytecode and executes it line by line. It follows the control flow and executes the instructions accordingly. Python follows indentation-based syntax to indicate the control flow structures.

  • Difference Between Class Attributes (cls) and Instance Attributes (self)

    • In Python, class attributes and instance attributes are both used to store data, but they differ in terms of their scope, usage, and lifetime.
      • Class Attributes (cls)
        • Shared by all instances of the class.
        • Defined at the class level and can be accessed or modified by any instance or method of the class.
        • Typically used for values or settings that should be consistent across all instances.
      • Instance Attributes (self)
        • Specific to each instance of the class.
        • Defined inside the __init__ method or other instance methods.
        • Each object (instance) has its own copy of the instance attributes, and modifying one instance's attribute does not affect others.
      • class Car:
	wheels = 4  # Class attribute (shared across all instances)

	def __init__(self, make, model):
		self.make = make  # Instance attribute (unique to each instance)
		self.model = model  # Instance attribute (unique to each instance)

	@classmethod
	def change_wheels(cls, new_wheel_count):
		cls.wheels = new_wheel_count  # Modify class attribute

	def display(self):
		print(f"Make: {self.make}, Model: {self.model}, Wheels: {self.wheels}")

# Create two instances
car1 = Car("Toyota", "Corolla")
car2 = Car("Honda", "Civic")

# Display instance data
car1.display()  # Make: Toyota, Model: Corolla, Wheels: 4
car2.display()  # Make: Honda, Model: Civic, Wheels: 4

# Modify class attribute via class method
Car.change_wheels(6)

# After modification, class attribute changes for all instances
car1.display()  # Make: Toyota, Model: Corolla, Wheels: 6
car2.display()  # Make: Honda, Model: Civic, Wheels: 6

# Modify instance attribute for car1
car1.wheels = 8  # This will not affect car2, as it's instance-specific

# Now car1 has a different number of wheels
car1.display()  # Make: Toyota, Model: Corolla, Wheels: 8
car2.display()  # Make: Honda, Model: Civic, Wheels: 6

  • What is the difference between .py and .pyc files?

    • .py files contain the source code of a program. Whereas, .pyc file contains the bytecode of your program. We get bytecode after compilation of .py file (source code). .pyc files are not created for all the files that you run. It is only created for the files that you import.

  • How do you copy an object in Python?

    • "=" operator does not create a copy instead it creates a binding reference. To copy we need to use copy module.
    • Shallow Copy is a bit-wise copy of an object. The copied object created has an exact copy of the values in the original object. If either of the values is a reference to other objects, just the reference addresses for the same are copied.
    • Deep Copy copies all values recursively from source to target object, i.e. it even duplicates the objects referenced by the source object.
    from copy import copy, deepcopy
list_1 = [1, 2, [3, 5], 4]
## shallow copy
list_2 = copy(list_1) 
list_2[3] = 7
list_2[2].append(6)
list_2    # output => [1, 2, [3, 5, 6], 7]
list_1    # output => [1, 2, [3, 5, 6], 4]
## deep copy
list_3 = deepcopy(list_1)
list_3[3] = 8
list_3[2].append(7)
list_3    # output => [1, 2, [3, 5, 6, 7], 8]

  • What is the difference between a shallow copy and a deep copy?

    • A shallow copy creates a new object and then copies the elements of the original object to the new object. However, if the elements of the original object are references to other objects, the shallow copy only copies those references, not the actual objects. As a result, both the original and the copied object will share the same internal references.
    • A deep copy, on the other hand, creates a completely independent new object and recursively copies all the elements and nested objects from the original object to the new one. In other words, it duplicates the entire object hierarchy, including all nested objects, so the copied object is fully independent of the original object.
    • Shallow copy can be done by copy.copy() while deep copy can be done by copy.deepcopy().

OOPS and Memory

  • Heap Allocation vs Stack Allocation vs Static Allocation

    • Heap Allocation is done using new keyword and Stack Allocation is done using local variables.
    • Stack memory is organized in a last-in-first-out (LIFO) structure called the stack.
    • Stack cannot persist beyond the lifetime of their function call. Once the function returns, the activation record is removed from the stack, and the memory used by local variables is released.
    • Heap memory is organized in a less structured manner. Memory is allocated from the heap and must be explicitly managed by the programmer.
    • Variables allocated on the heap have a longer lifetime than stack-allocated variables. They exist until explicitly deallocated by the programmer or until the program terminates. Calloc and Malloc are used for heap allocation. Free is used for deallocation.
    • The memory returned by calloc is initialized to zero for each byte, meaning it is effectively initialized to all 0's. The memory returned by malloc is not initialized, and it contains garbage values.
    • Heap memory is slower to allocate and deallocate than stack memory.
    • Stack memory is limited and typically smaller than heap memory.
    • Static Allocation does not makes data structures and objects dynamically like stack allocation.
    • It has fixed memory address and manual management (explicit initialization). Static allocation is not able to manage the allocation of memory at run time. Examples of static allocation are global and static variables.
